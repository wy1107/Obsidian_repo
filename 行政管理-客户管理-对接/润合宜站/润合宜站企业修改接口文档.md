## 1. 接口一：机构客户行政管理项目数据同步

### 1.1 接口功能描述

将外部行政管理项目传入的机构客户数据与系统正式表（FORMAL）中的现有数据进行合并，合并规则为：外部传入的非空字段覆盖正式表对应字段值。合并后的数据写入CHG变更表，同时生成一条业务申请记录（`BUSI_APPLY_INFO`），审批状态置为"待提交"（`to_submit`），等待后续审批流程。

**核心处理流程**：

1. 登录校验 — 校验当前用户是否已登录
    
2. 参数校验 — 校验请求体、`custNo`、`busiId` 是否为空
    
3. 客户存在性校验 — 通过 `custNo` 查询正式表 `CustMain`，确认客户存在
    
4. 查询正式表全量数据 — 查询该机构客户的所有关联正式表数据（26张子表）
    
5. 组装外部请求数据 — 将 `OrgCustAdminProjectInfo` 转换为 `OrgCustAddFullInfo` 格式
    
6. 数据合并 — 正式表数据 + 外部传入数据 → 合并数据（外部非空字段覆盖正式表）
    
7. 直传保护 — 对于外部未传入的MULTIPLE模式子表数据，从正式表直接镜像到CHG表，防止数据丢失
    
8. 写入CHG变更表 — 将合并后的全量数据写入CHG表
    
9. 保存业务申请信息 — 在 `BUSI_APPLY_INFO` 表中插入一条记录
    

### 1.2 请求方法

`POST`

### 1.3 请求URL

### 1.8 权限要求

- 需要登录认证（`Authorization` Header 携带有效 Token）
    
- 系统从登录信息中获取用户部门、用户编号、用户姓名等，自动填充到CHG变更表的创建人/更新人字段
    

### 1.9 注意事项

1. **合并策略**：外部传入的字段值非空时，将覆盖正式表中的对应字段；为空时保留正式表原有值。此为"外部数据优先"策略。
    
2. **CHG表机制**：数据不直接修改正式表，而是写入CHG变更表，需经过审批流程后方可生效。CustMain记录的审批状态初始为 `to_submit`（待提交）。
    
3. **数据保护**：对于外部未传入的MULTIPLE模式子表（如股东信息、高管信息、CRS信息等26张子表），系统会自动将正式表数据镜像到CHG表，避免合并时因未传字段导致数据丢失。
    
4. **事务性**：该接口使用 `@Transactional(rollbackFor = Exception.class)`，任一步骤异常将导致整个操作回滚。
    
5. **busiId来源**：建议先调用"判断C端客户是否有权限修改信息"接口获取由系统生成的 `busiId`，再将其传入本接口。
    
6. **ContentType**：本Controller类级别声明了 `consumes = {MediaType.APPLICATION_JSON_VALUE}`，请求体必须为JSON格式。

## 2. 接口二：判断C端客户是否有权限修改信息

### 2.1 接口功能描述

判断指定客户当前是否有权限进行信息修改。通过查询 `BUSI_APPLY_INFO` 表中是否存在该客户编号（`custNo`）对应且审批状态为"审批中"（`approval`）的记录来判断：

- **存在审批中的记录** → 返回失败，提示"此客户正在申请中，无法进行修改"，阻止重复提交
    
- **不存在审批中的记录** → 返回成功，使用雪花算法生成一个 `busiId` 返回给调用方，该 `busiId` 可用于后续的数据同步接口调用
    

### 2.2 请求方法

`GET`

### 2.3 请求URL

  
/ms-custmng/api/v1/admin-project/check-modify-permission?custNo={custNo}

### 2.4 请求参数列表

#### Query参数

|参数名|类型|必填|描述|
|---|---|---|---|
|custNo|String|是|客户编号，客户全局唯一ID|

### 2.5 返回参数说明

#### 2.5.1 返回结构

|参数名|类型|描述|
|---|---|---|
|code|int|响应状态码。200=成功（允许修改），500=失败（禁止修改）|
|msg|String|响应消息描述|
|data|Object|成功时返回雪花算法生成的busiId字符串；失败时为null|
|meta|GeneralMeta|元数据信息|

#### 2.5.2 成功响应示例（允许修改）

  
{  
    "code": 200,  
    "msg": "校验通过，允许修改",  
    "data": "1894567890123456789",  
    "meta": {}  
}

> `data` 字段值为雪花算法生成的业务申请号（`busiId`），可用于后续调用 `orgCustAdminProjectSync` 接口。

#### 2.5.3 失败响应示例（禁止修改）

  
{  
    "code": 500,  
    "msg": "此客户正在申请中，无法进行修改",  
    "data": null,  
    "meta": {}  
}

### 2.6 错误码定义

|错误码|含义|触发场景|
|---|---|---|
|200|成功|该客户不存在审批中的记录，允许修改|
|500|失败|该客户存在审批中的记录，禁止修改|

**失败场景明细：**

|场景|错误消息|
|---|---|
|客户存在审批中的申请|此客户正在申请中，无法进行修改|

### 2.7 调用示例

#### cURL 示例

  
curl -X GET 'http://localhost:9222/ms-custmng/api/v1/admin-project/check-modify-permission?custNo=C20260601000001' \  
  -H 'Authorization: Bearer {token}'

### 2.8 权限要求

- 需要登录认证（`Authorization` Header 携带有效 Token）
    

### 2.9 注意事项

1. **审批状态枚举**：`BUSI_APPLY_INFO` 表的 `APPROVE_STAT` 字段使用以下枚举值：
    
    - `to_submit` — 待提交
        
    - `approval` — 审批中
        
    - `finish` — 审批完成
        
    
    本接口仅校验"审批中"（`approval`）状态，"待提交"和"审批完成"状态不影响修改权限。
    
2. **busiId生成**：成功时返回的 `busiId` 由 `SnowflakeIdWorker`（雪花算法）生成，全局唯一，可用作后续调用 `orgCustAdminProjectSync` 接口的 `busiId` 参数。
    
3. **典型调用流程**：
    
    - 第一步：调用本接口校验权限并获取 `busiId`
        
    - 第二步：若校验通过，使用返回的 `busiId` 调用 `orgCustAdminProjectSync` 接口提交变更数据
        
4. **ContentType限制**：本接口为GET请求，不受Controller类级别 `consumes` 限制，但仍需在请求Header中携带 `Authorization`。

## 3. 通用说明

### 3.1 通用响应结构

所有接口统一使用以下响应格式：

  
{  
    "code": "int, 状态码",  
    "msg": "String, 响应消息",  
    "data": "Object, 响应数据",  
    "meta": "GeneralMeta, 元数据"  
}

### 3.2 审批状态流转

  
待提交(to_submit) → 审批中(approval) → 审批完成(finish)

- `checkCustModifyPermission` 接口会拦截 `approval` 状态的修改请求
    
- `to_submit` 和 `finish` 状态允许发起修改
    

### 3.3 双表机制说明

系统采用"正式表/变更表"双表机制：

|表类型|表名后缀|用途|操作权限|
|---|---|---|---|
|正式表|无后缀|存储当前生效的客户数据|审批通过后写入|
|变更表|CHG|存储待审批的客户变更数据|`orgCustAdminProjectSync` 写入此表|

数据变更流程：外部数据 + 正式表数据 → 合并 → 写入CHG表 → 审批通过 → 写入正式表
